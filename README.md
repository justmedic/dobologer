# dobologer

Движок сжатого блочного хранилища логов с полнотекстовым поиском на Rust.

Данные пишутся неизменяемыми блоками (по три файла на блок: `.meta` / `.data` / `.idx`).
Текст сжимается через `zstd` пачками, инвертированный индекс упаковывается через
`bitpacking`. Исторические блоки читаются через `mmap` без загрузки в память целиком:
в RAM постоянно живёт только активный буфер записи и сетевой порт.

## Возможности

- **Zero-RAM режим**: sealed-блоки лежат на диске, открываются через `mmap` по запросу.
- **ZSTD пачками**: текст режется на пачки по `PACK_ROWS` строк, каждая сжимается
  независимым фреймом. При поиске распаковывается только нужная пачка (~100 КБ), а не
  весь блок (10-50 МБ).
- **SIMD-индекс**: posting lists упаковываются через `BitPacker4x` (PFOR-Delta,
  чанки по 128 значений); короткие списки — varint-дельтами (~1–5 байт на ID).
- **O(1) ротация**: заполненный блок мгновенно вытесняется из активного буфера, а тяжёлый
  flush на диск идёт в фоне (`spawn_blocking`) без блокировки поиска/записи.
- **Без дыр видимости**: блок в процессе записи виден в поиске через промежуточный
  слой `flushing`.
- **Graceful shutdown**: по `Ctrl+C` / `SIGTERM` сервис дожидается завершения текущих
  запросов и дописывает активный (несброшенный) буфер на диск, чтобы логи не пропали.
- **TCP/UDP ingest**: приём логов по сети (порты 8081/8082), нарезка по `\n`, NDJSON-нормализация.
- **Булев поиск**: AND/OR/NOT через инфиксную строку (`GET`) или JSON-AST (`POST`).

## Требования

- Rust 1.85+ (edition 2021), `cargo`.
- Для `zstd` собирается системная C-библиотека (нужен C-компилятор, обычно уже есть).

## Сборка и запуск

```bash
# сборка
cargo build --release

# запуск (значения по умолчанию: ./data, 0.0.0.0:8080)
cargo run --release

# запуск бинарника
./target/release/dobologer
```

После старта в лог выводится:

```
dobologer listening on http://0.0.0.0:8080
```

Откройте в браузере `http://localhost:8080/` — встроенный веб-UI для поиска логов и
отправки тестовых записей через `/ingest`.

## Конфигурация (переменные окружения)

| Переменная              | По умолчанию   | Описание                                                        |
| ----------------------- | -------------- | -------------------------------------------------------------- |
| `DOBOLOGER_DATA_DIR`    | `./data`       | Каталог с блоками (`.meta`/`.data`/`.idx`).                     |
| `DOBOLOGER_BIND_ADDR`   | `0.0.0.0:8080` | Адрес и порт HTTP-сервера.                                      |
| `DOBOLOGER_BLOCK_ROWS`  | `65536`        | Сколько строк вмещает один блок до ротации на диск.             |
| `DOBOLOGER_PACK_ROWS`   | `1024`         | Сколько строк в одной независимой ZSTD-пачке внутри `.data`.    |
| `DOBOLOGER_SEARCH_LIMIT`| `1000`         | Дефолтный `limit` для `/search`, если параметр не передан. `limit=0` — без ограничения. |
| `DOBOLOGER_TCP_ADDR`    | `0.0.0.0:8081` | Адрес TCP-слушателя для приёма логов.                          |
| `DOBOLOGER_UDP_ADDR`    | `0.0.0.0:8082` | Адрес UDP-слушателя для приёма логов.                          |
| `DOBOLOGER_TCP_BATCH_LINES` | `1000`     | Макс. строк в TCP-буфере до сброса в движок.                    |
| `DOBOLOGER_TCP_FLUSH_MS`    | `300`      | Таймаут простоя TCP-буфера (мс) до сброса редких логов.        |

Пример запуска с кастомными настройками:

```bash
DOBOLOGER_DATA_DIR=/var/lib/dobologer \
DOBOLOGER_BIND_ADDR=127.0.0.1:9000 \
DOBOLOGER_BLOCK_ROWS=32768 \
cargo run --release
```

При старте сервис сканирует `DATA_DIR`, поднимает существующие блоки (по `.meta`) и
продолжает нумерацию с последнего `block_id`.

## Graceful shutdown

Сервис перехватывает `Ctrl+C` (SIGINT) и `SIGTERM`. При получении сигнала:

1. Прекращается приём новых соединений.
2. Дожидается завершения текущих in-flight запросов.
3. Дожидается завершения фоновых задач сброса блоков на диск.
4. Активный (ещё не заполненный) блок принудительно дописывается на диск.

Так буферизованные логи, не успевшие заполнить блок, не теряются: после перезапуска
сервис поднимет их с диска.

```bash
# корректно остановить
kill -TERM <pid>   # или Ctrl+C в терминале
```

В лог при этом выводится:

```
shutdown signal received, waiting for background flushes...
flushing active block...
active block flushed, exiting cleanly
```

> Примечание: аварийное завершение (SIGKILL, отключение питания) по-прежнему теряет
> активный буфер — в MVP нет WAL.

## HTTP API

Базовый адрес: `http://<host>:<port>` (по умолчанию `http://localhost:8080`).
Все тела запросов и ответов — в формате JSON.

### `POST /ingest`

Принимает батч строк логов и кладёт их в текущий активный блок. Когда блок заполняется
(`DOBOLOGER_BLOCK_ROWS`), он автоматически ротируется и **в фоне** сбрасывается на диск —
HTTP-ответ возвращается сразу, без ожидания zstd/bitpacking.

**Тело запроса** — JSON-массив строк:

```json
["auth_error: login failed", "user created successfully", "auth_error: token expired"]
```

**Ответ** `200 OK`:

```json
{ "ingested": 3 }
```

| Поле       | Тип     | Описание                              |
| ---------- | ------- | ------------------------------------- |
| `ingested` | integer | Количество принятых строк в этом запросе. |

**Пример:**

```bash
curl -X POST http://localhost:8080/ingest \
  -H 'Content-Type: application/json' \
  -d '["auth_error: login failed", "user created successfully"]'
```

**Коды ошибок:**

| Код   | Причина                                              |
| ----- | --------------------------------------------------- |
| `400` | Некорректный JSON или тело не является массивом строк. |

> Ошибки фонового сброса блока логируются в stderr и не влияют на ответ `/ingest`.

### `GET /search`

Ищет строки по булевому запросу (AND/OR/NOT). Опрашивает активный блок,
блоки в процессе записи (`flushing`) и все sealed-блоки на диске.

**Параметры запроса:**

| Параметр | Тип    | Обязательный | Описание                                       |
| -------- | ------ | ------------ | ---------------------------------------------- |
| `query`  | string | да           | Инфиксный булев запрос: термы, `AND`, `OR`, `NOT`, скобки. |
| `limit`  | integer | нет         | Максимум строк в `results`. По умолчанию **1000**. `limit=0` — вернуть все совпадения. Полное число — в `total`. |

**Синтаксис и приоритеты:**

- Операторы: `AND`, `OR`, `NOT` (регистронезависимо).
- Неявный `AND` между соседними термами: `auth error` == `auth AND error`.
- Приоритет: `NOT` > `AND` (явный и неявный) > `OR`.
- Пример: `auth OR error debug` == `auth OR (error AND debug)`.
- Термы нормализуются как при индексации: `env=prod` → `env AND prod`.
- Поиск регистронезависимый.
- Литералы `and`/`or`/`not` как термы — только через `POST /search` с JSON-AST.

**Ответ** `200 OK`:

```json
{
  "query": "auth_error AND NOT timeout",
  "count": 2,
  "total": 2,
  "results": [
    "auth_error: login failed",
    "auth_error: token expired"
  ]
}
```

| Поле      | Тип             | Описание                               |
| --------- | --------------- | -------------------------------------- |
| `query`   | string          | Исходный запрос (как передан клиентом). |
| `count`   | integer         | Число строк в `results` (≤ `limit`).   |
| `total`   | integer         | Полное число совпадений по всей базе.  |
| `results` | array of string | Оригинальные строки логов с совпадением. |

**Примеры:**

```bash
curl 'http://localhost:8080/search?query=error&limit=100'
curl 'http://localhost:8080/search?query=auth AND error'
curl 'http://localhost:8080/search?query=auth OR error debug'
curl 'http://localhost:8080/search?query=error AND NOT timeout'
```

**Примечания:**

- Порядок `results` не гарантируется между блоками (сначала active/flushing, затем sealed).
- Если совпадений нет — `count: 0`, `results: []`.

**Коды ошибок:**

| Код   | Причина                                          |
| ----- | ------------------------------------------------ |
| `400` | Отсутствует `query` или синтаксическая ошибка.   |
| `500` | Ошибка чтения/распаковки блока с диска.           |

---

### `POST /search`

Тот же поиск, но запрос передаётся как JSON-AST.

**Тело запроса:**

```json
{
  "query": {
    "and": [
      {"term": "auth"},
      {"not": {"term": "timeout"}}
    ]
  },
  "limit": 100
}
```

Формы узлов: `{"term":"x"}`, `{"and":[...]}`, `{"or":[...]}`, `{"not":{...}}`.

**Ответ** — как у `GET /search`; поле `query` содержит компактное строковое представление AST.

**Пример:**

```bash
curl -X POST http://localhost:8080/search \
  -H 'Content-Type: application/json' \
  -d '{"query":{"and":[{"term":"error"},{"not":{"term":"timeout"}}]},"limit":50}'
```

## TCP/UDP ingest

Помимо HTTP, логи можно отправлять по TCP (порт **8081**) и UDP (порт **8082**).
Строки режутся по `\n`. NDJSON нормализуется автоматически: JSON-строка → её содержимое;
JSON-объект → первое из полей `message`/`@message`/`msg`/`log`/`text`, иначе строка целиком.

**TCP** — потоковая передача, батчинг по размеру (1000 строк) или таймауту простоя (300 мс):

```bash
printf 'line one\nline two\n' | nc localhost 8081
echo '{"message":"user logged in"}' | nc localhost 8081
```

**UDP** — одна датаграмма может содержать несколько строк через `\n`:

```bash
echo -n 'udp log line' | nc -u localhost 8082
logger -n 127.0.0.1 -P 8082 -d 'syslog style message'
```

> UDP-датаграммы больше 64 КБ обрезаются. Буфер TCP при shutdown может не успеть сброситься.

## Быстрый старт (end-to-end)

```bash
# 1. Запустить сервер
cargo run --release &

# 2. Загрузить логи
curl -X POST http://localhost:8080/ingest \
  -H 'Content-Type: application/json' \
  -d '["GET /api/users 200", "POST /api/login 401 auth_error", "GET /api/users 500 db_error"]'

# 3. Найти строки по слову
curl 'http://localhost:8080/search?query=auth_error'
# {"query":"auth_error","count":1,"results":["POST /api/login 401 auth_error"]}

# 4. Булев поиск (пробелы в query — через --data-urlencode)
curl -G 'http://localhost:8080/search' --data-urlencode 'query=hello AND test'

# 5. TCP ingest
printf 'tcp log line\n' | nc localhost 8081
```

### Если порт занят (`Address already in use`)

Скорее всего остался фоновый процесс от `cargo run &`. Проверьте **8080, 8081, 8082** (не 8000):

```bash
lsof -i :8080 -i :8081 -i :8082
kill <PID>          # или: pkill dobologer
```

## Формат хранения на диске

Каждый блок — три файла `<block_id>.{meta,data,idx}`:

- **`.meta`** — JSON: `block_id`, `num_lines`, `rows_per_pack`, `data_size`, `idx_size`.
- **`.data`** — заголовок + `pack_offsets` + относительные `line_offsets` + независимые
  ZSTD-фреймы (по `PACK_ROWS` строк каждый).
- **`.idx`** — posting lists: полные блоки по 128 ID через `BitPacker4x`, короткие
  хвосты (<128) через varint-дельты; словарь термов + футер со смещением словаря.

## Тесты

```bash
cargo test
```

Покрывают токенизацию, булев поиск (AND/OR/NOT), NDJSON-нормализацию, TCP/UDP ingest,
кросс-блочный поиск, чтение sealed-блоков и видимость блоков в слое `flushing`.
Для быстрых тестов ротации можно уменьшить размер блока через `DOBOLOGER_BLOCK_ROWS`.

## Ограничения (MVP)

- `NOT` на больших блоках (~65k строк) возвращает крупные posting lists в памяти.
- Литералы `and`/`or`/`not` в инфиксном запросе интерпретируются как операторы.
- Активный блок дописывается при штатном завершении (SIGINT/SIGTERM), но теряется при
  аварийном (SIGKILL, отключение питания) — WAL нет. TCP-буфер при shutdown тоже может
  не успеть сброситься.
- Временные метки не хранятся: фильтрации по времени нет.
- UDP-датаграммы > 64 КБ обрезаются.
